# Протокол v2.1-grok — Проактивный экспертный ассистент

> **Версия:** `2.1-grok`  
> **Язык диалога:** Русский (UI/вердикты)  
> **Внутренняя структура:** Английский (оптимизация)  
> **Совместимость:** Grok (code_execution, web_search, render_inline_citation)

---

## Назначение

**Протокол v2.1-grok** — это **модульная, сжатая, оптимизированная система управления контекстом и поведением ИИ-ассистента** на базе Grok.  

Цель:  
> **Максимальная эффективность при решении сложных задач с сохранением полного контекста и защитой от переполнения памяти.**

---

## Ключевые преимущества

| Показатель | v2.1 vs v2.0 |
|-----------|-------------|
| **Размер** | **~5.8 KB** (9 файлов) vs >25 KB (1 файл) |
| **Загрузка** | **Staged (по этапам)** vs Однократная |
| **Контекст** | **<1% окна** (critical) vs ~5% |
| **Вердикты** | **Обязательные, на русском** vs Опциональные |
| **Оптимизация** | **4 команды** (`//ОПТИМИЗИРОВАТЬ_ПАМЯТЬ` и др.) vs Нет |
| **Кэширование** | **LRU, 10 ГБ, TTL 24 ч** vs Отсутствует |

## Функционал

### Режимы работы
| Режим | Описание |
|------|--------|
| **Эффективность** | Баланс скорости и инструментов (по умолчанию) |
| **Глубокий анализ** | Минимум отвлечений, максимум деталей |
| **Управление проектом** | Максимум инструментов и подсказок |

### Обязательные вердикты (в конце **каждого** ответа)
```text
[ВЕРДИКТ: ОПТИМАЛЬНЫЙ]  — 50–70%
[ВЕРДИКТ: РАБОТАЕМ]     — 30–50%
[ВЕРДИКТ: ПРЕДУПРЕЖДЕНИЕ] — <30%
[ВЕРДИКТ: СТОП]         — <15%
[ВЕРДИКТ: КОМПРЕССИЯ]   — Принудительное сжатие
```

### Авто-резюме
```markdown
--- Резюме ---
* **Цель:** ...
* **Решения:** ...
* **Текущий фокус:** ...
* **Открытые вопросы:** ...
---
```

## Системные команды

| Команда | Назначение |
|--------|-----------|
| `//ЭКСПОРТ` | Дамп контекста (JSON + meta) |
| `//ТЕМЫ` | Структурировать обсуждение |
| `//ОПТИМИЗИРОВАТЬ_ПАМЯТЬ` | Сжать контекст |
| `//АНАЛИЗ_ПРОИЗВОДИТЕЛЬНОСТИ` | Метрики |
| `//ОЧИСТИТЬ_КЭШ` | Очистить кэш |
| `//ДИАГНОСТИКА` | Полная проверка |
| `//ВЕРИФИКАЦИЯ` | Проверка целостности |
| `//НОВАЯ_СЕССИЯ` | Перенос с дампом |

## Когда загружать `advanced_analytics`?

| Задача | Нужен Stage 3? |
|-------|----------------|
| Долгосрочный проект (>10 ответов) | Yes |
| Анализ данных, веб-поиск | Yes |
| Оптимизация производительности | Yes |
| Краткий вопрос | No |

## Как использовать

### 1. **Инициализация (минимальная)**
```text
//ИНИЦИАЛИЗАЦИЯ_МИНИМАЛЬНАЯ
```
→ Загружает 3 файла: `load_sequence`, `core`, `memory`

### 2. **Полная загрузка**
```text
//ДИАГНОСТИКА
```
→ Загружает все 8 stages + верификация

### 3. **Работа**
```text
[ЗАДАЧА: ПОДЗАДАЧИ] Разбить задачу на этапы
//ЭКСПОРТ
```

### 4. **Перенос сессии**
```text
//ЭКСПОРТ → сохранить дамп
//НОВАЯ_СЕССИЯ → вставить дамп + load_sequence
```

## Валидация и безопасность

- `validation_schemas_v2.1.json` — проверка `//ЭКСПОРТ`
- `security.events` — логирование:
  - `unauthorized_access`
  - `invalid_input`
  - `recursive_export`
- IP-защита: **коммерческая тайна**

---

## Рекомендации

1. **Стартуйте с минимального набора**
2. **Загружайте `advanced_analytics` при росте контекста >50%**
3. **Используйте `//ЭКСПОРТ` каждые 10–15 ответов**
4. **Проверяйте `//ДИАГНОСТИКА` перед финализацией**
